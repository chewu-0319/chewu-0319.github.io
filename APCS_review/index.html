解題時建議一律將陣列宣告在 main 函式之上，
例如下列範例：

:::info
這樣的宣告方式會讓 price 陣列成為「全域變數」，
在大多時候全域變數是應該被避免使用的東西，
不過用在解題的領域相當方便，且副作用不大。寫大型應用程式再注意即可。
:::

:::warning
更正確的說是，不要宣告在任何函數內，包括但不限於 main()。
關於函數以及全域、區域變數的差異，在後續章節會再提及。
:::

結論而言，在 main() 裡面宣告太大的陣列（粗估 $10000$ 以上），
可能會在某些環境下導致程式當掉。

若宣告於 main() 之外，則宣告到一、兩千萬的程度通常沒什麼問題；
如果有問題，通常會在編譯時發生錯誤，而不是在執行後。

陣列的大小恆久不變

陣列一旦宣告完後，大小從此固定，無法改變。

如此一來，在宣告時由於仍無法得知所需個數，
必須考慮最差情形所需要的數量，再預留至少 5~10 作為緩衝。

:::info
一種思考方式是在輸入完後，依據輸入告知的數量決定陣列大小，
但也有不告知數量的題目；只要依最差情況多預留一些空間即可。
:::

例如題目最多給 1000 個整數，則宣告時至少需要 1005 的大小。
沒用上的部份就空著沒有關係，用來交換風險的降低非常划算。

一個用上緩衝的例子是：題目寫一寫發現，題目給的編號是從 1 開始計算；
為了可以使用索引 1~n，陣列大小就必須至少是 n+1 才行。
需要在頭尾各擺一道高牆之類的情形也不少見，預留緩衝可降低風險，
以免改了做法，卻忘了跟著改陣列大小。

像是月份就是從 1 開始，若只開剛好 12 的長度，就只能使用 0 到 11。

若題目未講明最大數值範圍，保險起見可以多宣告一些。

:::info
考慮一張照片大小約 2MB = 2,097,152 Bytes = 524,288 個 int 變數，
大概就能感受到，陣列宣告時其實不必太過節儉。
一、兩千萬個 int 通常都在可接受範圍內。
:::

字串

文字的處理鮮少只處理單一個文字，
因此需要能夠將多個文字視為一個單位的字串。

字串 string

:::info
字串為零或多個字元，按特定順序排列而成。
:::

最初的 cout << "hello world!\n" 的 "hello world!\n" 就是一個字串。
概念上和字元陣列有共通之處，但又不太相同。

字串型別與宣告

:::info
字串的型別為 string。
:::

例如以下程式碼示範如何宣告一個名為 str 的字串變數：

:::warning
和基本型別不同，字串是有包裝過的型別，犧牲一些效能換取便利性。
也因此，剛宣告的字串變數不同於其他型別，會被初始化為空字串。
:::

字串的常數

:::info
字串常數以雙引號 " 作為起始與結束。
:::

例如 "Hello World!\n" 就是一個字串常數。

:::warning
前面提過字串可以包含 0 個字元，也就是所謂的空字串。
用 "" 可以代表空字串。
:::

字串的輸入

字串型別同樣可直接透過 cin 輸入：

但文字和整數不同，整數可以很明確知道從哪裡開始、到哪裡結束，會是一個整數；
而基本所有輸入內容都可以作為字元看待，那麼請 cin 輸入一個字串時，
該到哪裡為止是一個段落，算是一個字串呢？

:::info
和輸入字元時相同，遇到 isspace 所認定的空白字元，
例如 tab、空白或者換行等，就會停下來，
將目前為止連續的非空白字元視為一個字串。

所有空白字元均會跳過，不會成為字串的內容。
:::

一個慣用技巧是：在字串的前後輸出中括號之類容易分辨的文字夾起來，
就能夠確認字串頭尾難以目視的空白、tab 或換行等空白字元。

試著輸入一些文字，夾雜各種符號或空白、換行等，觀察看看：

  • 哪裡到哪裡，會被認成一個字串？
  • 遇到哪些字元會被斷開？
  • 能斷開的那些字元，會不會被收錄到輸入的字串中？

:::warning
想想看，如果今天要處理的是一篇一般英文文章，
段落間的空白行或段落前的連續空白，該怎麼辦？
:::

以行為單位的輸入 getline

getline() 需要提供輸入的來源、以及存放字串的容器，
呼叫後便會輸入完整的一行，不會被空白字元截斷。

由於在 C++ 中，無號整數被認為比有號整數來得大，
因此有號整數在比較中，會先被轉換成無號整數，
而 -1 會轉換成 4294967295（$2^{32}-1$） 所以比較結果會是 false。
無號 32 位元整數無法儲存比它大的整數。
